TypeScript is the better long-term stack for this app.

Why:
- The app already has complex derived state: classes, backgrounds, spells, features, equipment, level-up, and rule-driven effects.
- Most bugs come from mismatched data shapes and duplicated logic between UI and rules.
- TypeScript helps keep the rule engine, character state, and creation flow consistent as the app grows.

Maintainability roadmap:
1. [ok] Move character-creation state logic fully out of app.js into TypeScript modules.
2. [ok] Create one typed CharacterService / CharacterEngine for derived calculations:
   - [ok] ability totals (src/core/character/ability-bonuses.ts)
   - [ok] spellcasting metrics (src/core/character/spell-engine.ts)
   - [ok] feature grants (src/core/character/feature-engine.ts)
   - [ok] equipment effects (src/core/character/inventory-engine.ts)
   - [ok] validation (src/core/state/creation-flow.ts)
3. [in progress] Keep app.js only as a render and wiring shell, then gradually replace it with TypeScript too.
4. [ok] Define stable TypeScript contracts for:
   - [ok] Character
   - [ok] BackgroundChoiceState
   - [ok] SpellChoiceState (as part of Character/Rules)
   - [ok] RuleDefinition
   - [ok] DerivedCharacterSheet
5. [pending] Add tests around behavior modules first, not the whole UI first.

Suggested execution order:
- Phase A: [ok] migrate the rest of builder and state logic from JS to TS
- Phase B: [ok] centralize all derived calculations in one typed engine
- Phase C: [in progress] make the UI consume only typed outputs (app.js is now a shell for these)
- Phase D: [pending] delete duplicated JS rule logic completely (clean up app.js wrappers)

Best next implementation target:
Migrate combat actions and effect engine (action-engine.js and modifier-engine.js) to TypeScript to complete the logic migration.
