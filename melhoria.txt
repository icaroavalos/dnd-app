TypeScript is the better long-term stack for this app.

Why:
- The app already has complex derived state: classes, backgrounds, spells, features, equipment, level-up, and rule-driven effects.
- Most bugs come from mismatched data shapes and duplicated logic between UI and rules.
- TypeScript helps keep the rule engine, character state, and creation flow consistent as the app grows.

Maintainability roadmap:
1. [ok] Move character-creation state logic fully out of app.js into TypeScript modules.
2. [in progress] Create one typed CharacterService / CharacterEngine for derived calculations:
   - ability totals
   - spellcasting metrics
   - feature grants
   - equipment effects
   - validation
3. Keep app.js only as a render and wiring shell, then gradually replace it with TypeScript too.
4. Define stable TypeScript contracts for:
   - Character
   - BackgroundChoiceState
   - SpellChoiceState
   - RuleDefinition
   - DerivedCharacterSheet
5. Add tests around behavior modules first, not the whole UI first.

Suggested execution order:
- Phase A: migrate the rest of builder and state logic from JS to TS
- Phase B: centralize all derived calculations in one typed engine
- Phase C: make the UI consume only typed outputs
- Phase D: delete duplicated JS rule logic completely

Best next implementation target:
Extract all spell, feature, background, and equipment derivation from app.js into a TypeScript character-engine module and make the UI read only from that.
