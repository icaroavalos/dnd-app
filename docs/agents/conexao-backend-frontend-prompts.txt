# Projeto de implementacao: frontend 100% conectado ao backend

Objetivo: transformar o backend NestJS na fonte obrigatoria de dados e regras do app, removendo fallback silencioso, hardcode e leitura direta de dados canonicos pelo frontend em runtime. Quando o backend falhar, a UI deve mostrar erro claro, bloquear a acao afetada e preservar estado sem inventar dados parciais.

Contexto obrigatorio para todos os agentes:
- Leia `docs/Architecture_memory.md`, `docs/preferences.md`, `docs/learnings.md`, `docs/sessions.md`, `docs/agents/README.md`, `docs/agents/task-board.md`, `docs/agents/roadmap.md`, `docs/superpowers/plans/2026-05-08-backend-mvp-system-roadmap.md`, `backend/README.md` e `backend/src/shared/CONTRACTS.md` se existir.
- Use Superpowers. Para implementacao, use TDD. Para bug ou comportamento inesperado, use systematic-debugging. Antes de concluir, use verification-before-completion.
- Nao edite `5etools-v2.28.0/` salvo quando a etapa mandar corrigir o compactador. O backend pode continuar usando `data/5etools/5e-2024/` e pipeline de compactacao como fonte interna; o frontend nao deve usar esses dados como fallback de runtime.
- Fallback silencioso e proibido no runtime do frontend para catalogos, projection, actions, resources, inventory e persistencia de personagens. Falha de backend deve virar estado de erro legivel na UI.
- `localStorage` pode continuar existindo para preferencias de UI e ultimo id ativo, mas nao como fallback silencioso para dados canonicos de ficha quando a tarefa de persistencia backend estiver concluida.
- Cada tarefa deve atualizar `docs/sessions.md` com timestamp, comandos rodados, arquivos alterados e resultado. Quando houver aprendizado tecnico ou decisao de arquitetura, atualize tambem `docs/learnings.md` e/ou `docs/Architecture_memory.md`.
- Cada tarefa deve fazer commit atomico. Nao misture frontend, backend e documentacao se o escopo puder ser separado.
- Comandos gerais esperados: `npm test`, `npm run typecheck`, `npm --prefix backend run test`, `npm --prefix backend run typecheck`. Para frontend local, suba backend e abra `http://127.0.0.1:4173` no Browser quando a etapa alterar UI.

Invariantes de aceite final:
- `rg -n "fallback|falling back|using fallback|local fallback|RACES.map|CLASSES.map|localStorage fallback" app.js src tests docs/agents` nao pode encontrar fallback silencioso em runtime do frontend; excecoes devem ser docs historicas ou testes de erro explicito.
- Criacao de personagem deve listar species/classes/backgrounds vindos do backend, inclusive dados compactados de `5etools` que ja estao corretos.
- Backend desligado deve produzir aviso/erro claro na interface, sem lista reduzida, mock, hardcode ou dados antigos mascarando a falha.
- Testes de contrato devem provar que frontend e backend concordam no shape de sucesso e erro.
